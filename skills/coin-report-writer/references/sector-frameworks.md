# Sector Frameworks

Use this file after resolving the asset identity and before drafting the report.

## Selection Rule

- Choose one `主赛道` and optionally one `次赛道`.
- State the chosen sector explicitly in the report.
- Weight the analysis according to the chosen sector instead of giving every section equal emphasis.
- If the asset spans multiple sectors, explain what actually drives valuation today rather than listing every possible label.

## Common Sectors

### 1. 比特币类货币资产

Use for BTC-like assets where the main thesis is monetary premium, store-of-value behavior, settlement role, treasury demand, or ETF flow sensitivity.

- Emphasize:
  monetary premium, scarcity, holder structure, ETF/fund flow, macro beta, miner economics, security budget
- Key metrics:
  circulating supply, issuance schedule, long-term holder data, miner reserves, ETF inflows/outflows, on-chain settlement value
- Common mistakes:
  overusing smart-contract or app-chain metrics that are not core to the thesis

### 2. L1 / L2 公链

Use for smart-contract platforms and scaling networks.

- Emphasize:
  developers, apps, users, fees, throughput, DA assumptions, decentralization, ecosystem stickiness
- Key metrics:
  active addresses, transactions, TVL, stablecoin balance, fees, DA usage, validator set, app diversity
- Common mistakes:
  confusing incentive-driven activity with durable adoption

### 3. DeFi 协议

Use for DEX, lending, derivatives, yield, stablecoin, or structured finance protocols.

- Emphasize:
  protocol revenue, token value capture, liquidity depth, composability, collateral quality, liquidation design
- Key metrics:
  TVL, volume, fees, revenue, take rate, open interest, borrower demand, reserve health
- Common mistakes:
  citing TVL alone without asking whether token holders capture any value

### 4. AI / 计算 / 数据网络

Use for AI infra, decentralized compute, data labeling, agent platforms, or AI-related protocols.

- Emphasize:
  real AI utility, paying demand, supply-side economics, compute/data bottlenecks, moat versus centralized incumbents
- Key metrics:
  paying customers, workload volume, compute supply, model/data usage, token incentives, gross margin analogs
- Common mistakes:
  confusing AI narrative premium with actual product demand

### 5. DePIN / 物理基础设施

Use for wireless, sensor, mapping, GPU, storage, energy, or hardware-backed token networks.

- Emphasize:
  hardware deployment, utilization, unit economics, geographic density, service quality, incentive sustainability
- Key metrics:
  active devices, coverage/utilization, revenue, cost to acquire operators, emissions versus revenue, geography mix
- Common mistakes:
  celebrating node count without verifying real demand or utilization

### 6. 中间件 / 预言机 / 跨链 / 数据可用性

Use for oracle, interoperability, messaging, rollup infra, DA, indexing, or backend middleware assets.

- Emphasize:
  integration depth, switching cost, dependency centrality, reliability, security assumptions, customer concentration
- Key metrics:
  integrations, secured value, query volume, chain coverage, uptime, fee volume, dependency share
- Common mistakes:
  counting integrations equally when many are low-value or inactive

### 7. Meme / 社区驱动代币

Use when culture, attention, exchange support, and liquidity reflexivity matter more than product fundamentals.

- Emphasize:
  social attention, holder growth, exchange listings, liquidity, whale concentration, cycle sensitivity
- Key metrics:
  holder count, social engagement, exchange depth, perpetual OI, funding, concentration, velocity
- Common mistakes:
  pretending there is strong fundamental value capture when the main driver is attention

### 8. RWA / 支付 / 合规金融轨道

Use for tokenized treasuries, payment rails, yield-bearing stable structures, or compliance-heavy assets.

- Emphasize:
  legal structure, reserve quality, counterparties, licensing, redemption mechanics, institutional distribution
- Key metrics:
  AUM, issuance/redemption, reserve composition, partner quality, jurisdiction, fee model, concentration risk
- Common mistakes:
  ignoring legal and counterparty structure while focusing only on on-chain growth

### 9. 交易所 / 钱包 / 消费平台代币

Use for exchange tokens, wallet ecosystem tokens, app-platform tokens, or consumer-distribution plays.

- Emphasize:
  user distribution, platform revenue, token utility, buyback/burn design, compliance exposure, market-share durability
- Key metrics:
  users, trading volume, revenue, market share, burn amount, app usage, regional dependence
- Common mistakes:
  overvaluing token utility when the platform can grow without meaningful token capture

### 10. 股票 / 上市公司

Use for publicly listed companies, ADRs, or equities where the core object is a business plus an exchange-traded share.

- Emphasize:
  revenue quality, growth durability, gross margin, operating leverage, cash flow conversion, balance sheet, dilution or buybacks, valuation multiple, management execution
- Key metrics:
  market cap, enterprise value, revenue, growth, gross margin, operating margin, EPS, free cash flow, net cash or net debt, diluted share count, SBC, buyback pace, peer multiples
- Common mistakes:
  forcing on-chain or tokenomics metrics onto equities, ignoring dilution, or comparing against famous stocks that do not share the same business model

## Writing Rule

- In `关键指标快照`, choose metrics that fit the chosen sector.
- If the asset is a stock, prefer equity metrics over crypto-native metrics and explain why the current multiple is or is not justified.
- In `全方位多角度分析`, use the chosen sector to decide which angles should be marked `强`, `中性`, `弱`, or `不适用`.
- In `前景分析`, compare the asset against the closest peers in the same sector, not random large-cap tokens.
